# 技能制作方法论

赶时间？跳到底部检查清单，逐条过。

---

## 核心原则

**假设是最大的敌人**：第一版必然基于假设，代码审查看不出来（逻辑自洽），只有真实测试才能暴露。

**发现 > 执行**：好的技能帮用户看见盲区，而不是执行已知操作。用户自己说不全需求——看到结果后才想起遗漏。

**做减法**：AI 倾向于加功能、加自动化、加配置。用户几乎总在纠正"这个不用管"、"让人自己装"。只管容易忘/容易漏的。

**修根因**：同一类问题出现第三次就该重构，别继续打补丁。

**通用化**：用日常语言写，所有人都能用。用术语写，只有你能用。

---

## 全流程总览

```
Phase 1: 问题建模（10 分钟，省 3 小时）
    ↓
Phase 2: 草稿 —— 三阶段架构 + 契约 + 最小去重/过滤
    ↓
Phase 3: 真实测试（最重要）
    ↓
Phase 4: 审计分级 —— P0/P1/P2，修根因不修症状
    ↓
    └──→ 回到 Phase 3，直到 P0/P1 全清
    ↓
Phase 5: 做减法
    ↓
Phase 6: 通用化
    ↓
Phase 7: 契约同步（配对技能）
    ↓
检查清单自检
```

## Phase 1：问题建模

写代码之前回答四个问题。不超过 10 行。

### 1. 一句话说清要解决什么问题

超过一句说明还没想清楚。需要解释背景才能说清 = 还没抓住核心。

### 2. 环境差异检查

逐项过，每项写"有影响"或"无影响（理由）"：

| 维度 | 检查项 |
|------|--------|
| 路径 | 用户目录会被重定向吗？数据在多个盘上吗？ |
| 编码 | 路径和文件名会有非 ASCII 字符吗？ |
| 数据量 | 最少几条？最多几十万条？性能会崩吗？ |
| 权限 | 需要管理员权限吗？文件会被锁定吗？ |
| 平台 | 只在某个系统？需要特定软件？ |
| 配对 | 有没有另一个技能会消费这个技能的输出？ |

列不出差异 = 对问题空间理解不够。

### 3. 数据契约（配对技能才需要）

```json
{
  "_contract": {
    "writer": "技能A",
    "reader": "技能B",
    "version": 1,
    "required_fields": [],
    "optional_fields": []
  }
}
```

要点：写明谁写谁读、每个字段有含义、有版本号、改了生产者必须同步消费者。

### 4. 静默失败场景

什么情况下不报错但结果错误？这是最危险的 bug。

常见模式：硬编码路径指向空目录、退出码判断不对、编码问题导致文件列表为空、过滤条件太激进。

## Phase 2：草稿

### 三阶段架构

```
发现（Discovery）→ 只回答"有什么"，不做过滤
    ↓ 原始数据列表
处理（Processing）→ 去重/过滤/分类，只回答"该留什么"
    ↓ 干净数据列表
执行（Execution）→ 复制/安装/生成，只回答"怎么做"
```

每阶段输入是上一阶段的输出。跳步 = 混入脏数据。

### 去重和过滤

第一版不用完美，但要**架构预留**——有去重步骤、有过滤步骤，实现可以简单。

去重最小实现：路径归一化（去末尾斜杠、统一大小写）+ 去重集合。
过滤最小实现：先硬编码排除列表（node_modules、.git、Cache 等），后面改可配置。

### 环境感知

用 API/命令探测实际环境，不硬编码。坏例子：假设桌面在用户目录下。好例子：用系统 API 查实际路径。坏例子：假设只有 C 盘。好例子：扫描所有本地盘。

### 体量控制

**SKILL.md 不超过 500 行。** 超过就提取：代码片段（>20 行）→ `scripts/`，参考信息 → `references/`。

拆成两个技能的信号：触发条件不同、可独立使用、合在一起超 500 行。

### Description 怎么写

写清楚**做什么 + 什么时候用**。触发场景列具体。宁可多触发不要漏触发——触发错了可以跳过，漏触发了用户不知道这个技能存在。

---

## Phase 3：真实测试

**最重要的阶段。** 代码审查发现逻辑错误，真实测试发现假设错误。

### 测试前

1. 记录当前环境状态，测完能对比
2. 备份当前数据，测试可能破坏环境
3. 明确预期（数据量、文件数、输出格式）

### 测试方法

用 Agent 开一个子对话来跑，不带当前上下文，模拟"新用户拿到技能"的体验。

**看什么**（按优先级）：

| 检查项 | 通过标准 |
|--------|---------|
| 数据量/规模 | 与独立估算对比，误差不超 30% |
| 重复 | 零重复 |
| 遗漏 | 核心数据不遗漏（用手动基线对比） |
| 垃圾 | 无缓存、临时文件、无关数据混入 |
| 报错 | 零 P0/P1 错误 |

查遗漏不能只看输出——你不知道你遗漏了什么。用手动基线对比。

### 退出标准

- 没有 P0（数据丢失/损坏）
- P1 已记录，有修复方向
- 可复现（幂等性）

### 典型问题速查

| 症状 | 根因 | 修复方向 |
|------|------|---------|
| 数据量远小于预期 | 硬编码路径 | 改用系统 API |
| 数据量远大于预期 | 包含缓存/临时文件 | 加过滤 |
| 有重复 | 去重没做 | 加路径归一化 |
| 静默失败 | 退出码判断不对 | 检查错误处理 |
| 结果不稳定 | 环境相关 | 加环境检测 |
| 不幂等 | 状态累积 | 每次从干净状态开始 |

---

## Phase 4：审计与分级

- **P0**（数据丢失/损坏）：立即修复，不能往下走。
- **P1**（功能错误）：修完再测。
- **P2**（体验问题）：P0/P1 修完再处理。

**一轮修一类问题**，修完立即重测。修一个问题引入新问题，或同一类问题出现第三次 → 停下来重构。

---

## Phase 5：做减法

功能正确后先删：
- AI 想加但用户不需要的步骤
- 只有极端场景才用到的逻辑
- 需要长篇解释的设计（说明设计不够直觉——改设计，别加解释）

判断标准：删掉这一步，最常见的用户会出什么问题？答"不会出问题"就删。

---

## Phase 6：通用化

- 技术术语 → 日常用语
- 配置选项 → 自动检测
- 专家判断 → 直觉规则

判据：非技术朋友能理解每一步在干什么吗？

---

## Phase 7：契约同步

配对技能必须：
- 改了一边立即更新另一边
- 测试时两端一起测
- 有版本号，旧格式走降级逻辑

---

## 错误处理

**发现阶段**：路径不存在 → 跳过并记录；权限不足 → 告知用户，不静默跳过。

**处理阶段**：数据为空 → 标注"未发现"；全部被过滤 → 问用户是否调整。

**执行阶段**：失败 → 原因 + 行动建议；部分失败 → 列出成功和失败的。

给用户的错误信息三要素：出了什么问题 + 为什么 + 用户可以做什么。

---

## 常见陷阱

1. 只做代码审查不跑真实测试
2. 假设"标准环境"——没有标准环境
3. AI 做加法，用户做减法
4. 修症状不修根因——第三次就该重构
5. 跳过过滤设计——"不漏"不够，还要"不冗余"
6. 技术语言陷阱——术语只有你能用
7. 配对不同步——改了生产者忘了消费者
8. 跳过 Phase 1——没想清楚就动手，写到一半推翻重来
9. 发现和执行混在一起——改一处坏三处
10. 数据发现是增量的——用户说不全需求，设计要支持增量发现

## 检查清单

### 架构
- [ ] 三阶段分离（发现 → 处理 → 执行，可独立运行和测试）
- [ ] 无硬编码，用 API/命令探测实际环境
- [ ] 去重逻辑已包含
- [ ] 过滤逻辑已包含

### 契约（如适用）
- [ ] 配对技能的数据契约已定义（格式、字段、版本）
- [ ] 契约中有 `_contract` 或等价元信息
- [ ] 改了一端，另一端已同步

### 测试
- [ ] 用真实数据测试过
- [ ] 数据量/质量符合预期（误差不超 30%）
- [ ] 测试可复现（幂等性）
- [ ] 环境差异已覆盖

### 通用性
- [ ] 非技术人员能理解每一步
- [ ] 没有只有开发者才能理解的配置
- [ ] 失败提示包含：问题 + 原因 + 行动建议

### 体量
- [ ] SKILL.md 不超过 500 行
- [ ] Description 写了做什么 + 什么时候触发
