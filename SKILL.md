# 技能制作方法论

从 windows-backup 的创建过程中提炼。那个技能从一个 7GB 的残缺备份，
经过十轮迭代（4 次上下文溢出，跨 12 小时），最终产出 32.9GB 的干净完整备份。

赶时间？跳到底部检查清单，逐条过。

---

## 设计哲学

### 假设是最大的敌人

你写的第一版必然基于假设。这些假设在你看不到的地方是错的。

windows-backup 假设桌面在 `C:\Users\用户名\Desktop`——实际上 Windows 允许把桌面
重定向到任何位置。7GB vs 30GB 的差距不是代码 bug，是假设错误。
代码审查看不出这个问题（代码逻辑自洽），只有真实测试才能发现。

### 发现 > 执行

windows-backup 最有价值的不是 robocopy（谁都会写），是"帮你找出你不知道自己有的东西"。
用户自己也无法一次性说出所有需要备份的东西——是看到一部分结果后才想起另一部分。
**好的技能帮用户看见盲区，而不是帮用户执行已知操作。**

### 做减法，不做加法

AI 倾向于做加法：覆盖更多场景、自动化更多步骤、加更多配置项。
用户的纠正几乎全是做减法：

| AI 想做的 | 用户纠正 |
|-----------|---------|
| 自动装 NVM、Node.js | "装不了，我实际操作过" |
| 自动装 Chrome、微信 | "基础工具让人自己装" |
| 覆盖驱动安装 | "懂的人不需要教，不懂的人学了也用不上" |
| 面向开发者 | "只要认识中文就能用" |

做减法比做加法更难，但更有价值。只管容易忘/容易漏的东西。

### 根因胜于打补丁

测试发现问题后，修根本原因。windows-backup 的根因不是"漏了几个路径"，
而是"路径发现机制本身有问题"。从硬编码改成系统 API 查询，一次修好所有路径问题。
六个补丁之后你就不敢改了——别走到那一步。

### 通用胜于精确

"视频、图片、文档"比".mp4, .jpg, .docx"更多人能看懂。
通用化不只是去掉硬编码，更深层的是**语言和心智模型的通用化**。
技术人默认用技术语言思考，面向普通用户必须换成日常语言。

## 全流程总览

```
Phase 1: 问题建模 —— 想清楚再动手（10 分钟）
    ↓
Phase 2: 草稿 —— 三阶段架构 + 契约 + 最小去重/过滤
    ↓
Phase 3: 真实测试 —— 在真实环境跑一遍（最重要）
    ↓
Phase 4: 审计分级 —— P0/P1/P2，修根因不修症状
    ↓
    └──→ 回到 Phase 3，直到 P0/P1 全清
    ↓
Phase 5: 做减法 —— 删掉用户不需要的
    ↓
Phase 6: 通用化 —— 换成日常语言
    ↓
Phase 7: 契约同步 —— 配对技能一起改一起测
    ↓
检查清单自检
```

## Phase 1：问题建模

写代码之前，回答这四个问题。不超过 10 行。花 10 分钟，省 3 小时。

### 1. 一句话说清要解决什么问题

超过一句说明问题还没想清楚。如果需要解释背景才能说清，
说明你还没抓住问题的核心。

### 2. 环境差异最小检查表

逐项过，每项写"有影响"或"无影响（理由）"：

| 维度 | 检查项 |
|------|--------|
| 路径 | 用户目录会被重定向吗？数据会在多个盘上吗？ |
| 编码 | 路径和文件名会有非 ASCII 字符吗（中文、日文、空格）？ |
| 数据量 | 最少几条？最多几十万条？性能会崩吗？ |
| 权限 | 需要管理员权限吗？文件会被其他进程锁定吗？ |
| 平台 | 只在 Windows？只在线？需要浏览器？需要特定软件？ |
| 配对 | 有没有另一个技能会消费这个技能的输出？ |

你列不出差异，说明你对问题空间的理解还不够。

### 3. 数据契约

如果有配对技能（备份/恢复、生成/部署），定义中间格式。

最小契约模板：
```json
{
  "_contract": {
    "writer": "技能A的名字",
    "reader": "技能B的名字",
    "version": 1,
    "required_fields": ["field1", "field2"],
    "optional_fields": ["field3"]
  },
  "field1": "值",
  "field2": "值"
}
```

契约要点：
- 写明谁写、谁读
- 每个字段有含义说明
- 有版本号，格式变了走降级逻辑
- 改了生产者必须同步消费者

### 4. 静默失败场景

什么情况下技能不报错但结果错误？这是最危险的 bug。

常见静默失败：
- 硬编码路径指向了空目录（没报错，但什么都没找到）
- 进程退出码判断不对（robocopy 返回 0-7 都是正常，8+ 才是错误）
- 编码问题导致中文路径读不到文件（没报错，但文件列表为空）
- 过滤条件太激进，把有用的东西过滤掉了（没报错，但结果不完整）

## Phase 2：草稿

### 三阶段架构

```
发现阶段（Discovery）
  扫描/收集/探测，只回答"有什么"
  不做任何过滤或判断
      ↓ 输出：原始数据列表
处理阶段（Processing）
  去重/过滤/分类/计算，只回答"该留什么"
  不执行任何实际操作
      ↓ 输出：干净数据列表
执行阶段（Execution）
  复制/安装/配置/生成，只回答"怎么做"
  只操作处理阶段的输出，不回头找原始数据
```

每阶段的输入是上一阶段的输出。跳步 = 混入脏数据。

### 第一版去重和过滤

第一版的去重/过滤不用完美，但要**架构预留**：

去重最小实现（先跑通，后面加强）：
```powershell
# 路径归一化：去掉末尾斜杠，统一大小写
$normalized = $path.TrimEnd('\').ToLower()
# 用 HashSet 检测重复
$seen = @{}
if ($seen[$normalized]) { Write-Output "跳过重复: $path"; continue }
$seen[$normalized] = $true
```

过滤最小实现：
```powershell
$excludePatterns = @("node_modules", ".git", "Cache", "*.tmp")
# 第一版先硬编码这些，后面改成可配置
```

第一版和"做完再加"的区别：**架构上预留了位置**（有去重步骤、有过滤步骤），
只是实现可以简单。后面加强时不用重构架构。

### 环境感知

用 API/命令探测实际环境，不用硬编码。

```powershell
# 坏：假设桌面在用户目录下
$desktop = "$env:USERPROFILE\Desktop"

# 好：问系统实际路径在哪
$desktop = [Environment]::GetFolderPath("Desktop")

# 坏：假设所有盘都是 C:
# 好：扫描所有本地盘
$allDrives = Get-Volume | Where-Object { $_.DriveLetter -and $_.DriveType -eq 'Fixed' }

# 坏：假设路径中只有 ASCII 字符
# 好：文件操作统一用 UTF-8 编码
Get-Content $file -Raw -Encoding UTF8
```

### Scope 和体量控制

**SKILL.md 控制在 500 行以内。** 超过就把代码提取到 `scripts/`，参考资料提取到 `references/`。

判断标准：
- 代码片段超过 20 行 → 考虑提取到 `scripts/`
- 参考信息（平台规则、API 文档、配置模板）→ 放 `references/`，SKILL.md 里写"详见 references/xxx.md"
- 多个技能共享同一套逻辑 → 抽出公共 `references/` 文件

什么时候拆成两个技能：
- 两个技能的触发条件完全不同（"备份"和"恢复"）
- 两个技能可以独立使用（不需要另一个才能工作）
- 合在一起超过 500 行且没有强关联

### Description 怎么写

Description 是触发机制——写得好技能才会在对的时候被使用。

写法要点：
- **说清楚做什么 + 什么时候用**：两个都要有
- **触发场景列具体**：不只说"创建技能"，还要说"技能总是出问题"、"怎么写好 skill"
- **稍微"推"一点**：宁可多触发，不要漏触发。触发错了可以跳过，漏触发了用户不知道这个技能存在

好的 description：
> 当用户要求创建新技能、改进现有技能、或调试技能质量问题时触发。
> 也适用于"怎么写好一个 skill"、"技能总是出问题"的场景。

不好的 description：
> 创建技能的方法论。

## Phase 3：真实测试

**这是整个流程中最重要的阶段。** 代码审查发现逻辑错误，真实测试发现假设错误。
windows-backup 的代码审查全部通过，但模拟测试才发现只有 7GB。

### 测试前准备

1. **记录当前环境状态**：测试前截图或记录关键数据，测完能对比
2. **备份当前数据**：测试可能破坏环境，先保底
3. **明确预期**：写下你认为测试结果应该是什么样的（数据量、文件数、输出格式）

### 测试方法

**怎么测**：用 Agent 开一个子对话来跑技能。这个子对话不带当前对话的上下文，
能更真实地模拟"一个新用户拿到这个技能"的体验。

**看什么**（按优先级）：

| 检查项 | 怎么判断 | 通过标准 |
|--------|---------|---------|
| 数据量/规模 | 对比你独立估算的真实数据量 | 误差不超过 30% |
| 有没有重复 | 同一文件/数据出现两次 | 零重复 |
| 有没有遗漏 | 对照你知道应该有的数据 | 核心数据不遗漏 |
| 有没有垃圾 | 缓存、临时文件、无关数据混入 | 无明显垃圾 |
| 有没有报错 | 运行过程中是否有错误 | 零 P0/P1 错误 |

**查遗漏的方法**：先用手动方式或另一个已知可靠的工具拿到一份基线，然后对比。
你不知道你遗漏了什么——所以不能只靠"看输出"，要靠"对比"。

### 测试退出标准

一轮测试可以进 Phase 4 的条件：
- 没有 P0（数据丢失/损坏）
- P1 问题已记录，有修复方向
- 测试结果可复现（跑两遍结果一样——幂等性）

### 测试中发现的典型问题

| 症状 | 可能的根因 | 修复方向 |
|------|-----------|---------|
| 数据量远小于预期 | 硬编码路径，漏掉重定向的文件夹 | 改用系统 API 查路径 |
| 数据量远大于预期 | 包含了缓存/临时文件 | 加过滤，排除 Cache 等 |
| 有重复数据 | 路径去重没做 | 加路径归一化和去重 |
| 静默失败 | 错误被吞，或退出码判断不对 | 检查错误处理逻辑 |
| 结果时对时错 | 环境相关，某些条件下才触发 | 找到触发条件，加环境检测 |
| 跑两次结果不一样 | 状态累积或时间依赖 | 加幂等性：每次从干净状态开始 |

## Phase 4：审计与分级

测试发现的问题，按严重程度分级：

- **P0（数据丢失/损坏）**：用户会因此丢东西。立即修复，不能往下走。
- **P1（功能错误）**：功能不工作，但不丢数据。修完再测试。
- **P2（体验问题）**：能用但不舒服。P0/P1 修完再处理。

### 修复原则

- **先修根因，不修症状。** 症状是"数据量不够"，根因是"路径发现机制有问题"。
- **一轮修一类问题。** 修完立即重新测试，确认没引入新问题。
- **架构判断标准**：如果修一个问题引入了新问题，或者同一类问题出现第三次，
  说明不是个别 bug，是架构有问题。停下来重构，别继续打补丁。

---

## Phase 5：做减法

功能正确后，先别急着优化，先删：

- **删掉 AI 想加但用户不需要的步骤。** AI 会想自动装软件、自动配置环境——
  基础工具让人自己装，AI 只管容易忘/容易漏的东西。
- **删掉只有极端场景才用到的逻辑。** "懂的人不需要教，不懂的人学了也用不上"——
  这种逻辑不值得维护。
- **删掉多余的解释。** 如果某个步骤需要长篇解释"为什么这样设计"，
  说明设计还不够直觉——改设计，别加解释。

**怎么判断该不该删**：问自己"如果我删掉这一步，最常见的用户会出什么问题？"
如果答案是"不会出什么问题"，就删。

---

## Phase 6：通用化

做减法之后，换语言：

- 技术术语 → 日常用语（".mp4/.avi/.mkv" → "视频文件"）
- 配置选项 → 自动检测（不问用户"你的桌面在哪"，直接问系统）
- 专家才能理解的判断 → "大 + 旧"式直觉规则（不问用户"要不要排除 .cache 目录"，
  改为"超过 6 个月没动过的大文件，要不要排除？"）

**通用化判据**：给一个非技术朋友看，他能理解每一步在干什么吗？
如果不能，换说法。

---

## Phase 7：契约同步

如果技能有配对（如备份+恢复、生成+部署）：

- 改了一边的输出格式，**立即**更新另一边。不要攒着。
- 测试时两端一起测：用 A 产出的数据给 B 消费，看 B 能不能正确处理。
- 在契约文件里写明版本号。旧格式走降级逻辑，不能直接崩溃。

## 错误处理原则

每个阶段都应该考虑的错误处理：

**发现阶段**：
- 扫描到的路径不存在 → 跳过并记录"跳过了什么"
- 权限不足读不了 → 记录并告知用户，不要静默跳过

**处理阶段**：
- 数据为空 → 不是错误，但输出要标注"未发现 XX 数据"
- 去重后全部被过滤 → 告知用户"所有数据都被过滤了，是否调整过滤条件"

**执行阶段**：
- 操作失败 → 告知用户失败原因 + 下一步可以做什么（不只是"失败了"）
- 部分失败 → 告知哪些成功了、哪些失败了，不要全有或全无

**给用户的错误信息**包含三个部分：
1. 出了什么问题（事实）
2. 为什么出问题（原因，用日常语言）
3. 用户可以做什么（行动建议）

## 常见陷阱

1. **只做代码审查不跑真实测试**：审查看的是逻辑，测试看的是现实。
   windows-backup 代码审查全通过，模拟测试才发现只有 7GB。
2. **假设"标准环境"**：没有标准环境。你的机器是你的假设，别人的机器不是。
3. **AI 做加法，用户做减法**：AI 想覆盖更多场景、自动化更多步骤。
   用户几乎总在纠正"这个不用管"、"让人自己装"、"不懂的人用不上"。
4. **修症状不修根因**：每次打补丁，下次改更难。同一类问题出现第三次就该重构。
5. **跳过过滤设计**：没有过滤的技能会把垃圾当宝贝。"不漏"不够，还要"不冗余"。
6. **技术语言陷阱**：用术语写的东西只有你能用，用日常语言写的所有人都能用。
7. **配对不同步**：改了生产者忘了消费者，数据契约名存实亡。
8. **跳过 Phase 1**：没想清楚就动手写，写到一半发现方向错了，全部推翻重来。
9. **发现和执行混在一起**：改一处坏三处。发现阶段发现不了的东西，执行阶段再怎么补也补不回来。
10. **数据发现是增量的**：用户自己也无法一次说完所有需求，看到结果后才想起遗漏。
    技能设计要支持增量发现，不是"一次性问完"。

## 检查清单

写完技能后逐条过。一条不过就说明还有问题。

### 架构
- [ ] 三阶段分离：发现 → 处理 → 执行（每阶段可以独立运行和独立测试）
- [ ] 没有硬编码路径，用 API/命令探测实际环境
- [ ] 去重逻辑已包含（有明确的去重步骤，不是事后补丁）
- [ ] 过滤逻辑已包含（有排除列表，列表非空或有注释说明为什么不需要排除）

### 契约（如适用）
- [ ] 配对技能的数据契约已定义（格式、字段、版本）
- [ ] 契约中有 `_contract` 或等价的元信息（谁写、谁读、版本号）
- [ ] 改了一端，另一端已同步

### 测试
- [ ] 用真实数据测试过（数据量级接近真实场景的 10% 以上）
- [ ] 测试结果的数据量/质量符合预期（与独立估算对比，误差不超 30%）
- [ ] 测试可复现（跑两遍结果一样——幂等性）
- [ ] 环境差异已覆盖（对 Phase 1 列出的每项差异，标注了：已测试 / 风险低可跳过+理由）

### 通用性
- [ ] 语言通用，给非技术朋友看他也能理解每一步在干什么
- [ ] 没有只有开发者才能理解的配置
- [ ] 失败时的提示包含：出了什么问题 + 为什么 + 用户可以做什么

### 体量
- [ ] SKILL.md 不超过 500 行（超过则提取代码到 scripts/、参考到 references/）
- [ ] Description 写了做什么 + 什么时候触发（两条都有）
